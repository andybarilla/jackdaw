# Pi workbench prototype validation checklist

Use this after each monitoring session and again at the end of the one-week trial.

## Core success checks

- [ ] I monitored 3-5 active sessions from one overview screen.
- [ ] I identified the session needing attention in under 10 seconds.
- [ ] I handled routine status checks from summary + status + tool activity without opening raw transcript details at least 80% of the time.
- [ ] `steer` was reachable through an obvious path.
- [ ] `follow-up` was reachable through an obvious path.
- [ ] `abort` was reachable through an obvious path.
- [ ] pinned summaries were usable in normal flow.
- [ ] editable session names were usable in normal flow.
- [ ] session tags were usable in normal flow.

## Before vs. current Jackdaw

- [ ] I checked fewer raw transcripts than I normally would in current Jackdaw.
- [ ] I intervened faster than I normally would in current Jackdaw.
- [ ] structured state felt more useful than watching terminal panes.

## Shell fallback

- [ ] shell fallback was not needed for routine monitoring.
- [ ] when shell fallback was needed, the reason was documented in the usage log.
- [ ] returning from shell fallback back to the dashboard felt clear.

## End-of-week decision

- [ ] enough evidence was gathered to choose: continue, iterate, or stop.
- [ ] the decision memo was updated with specific examples, not just impressions.
