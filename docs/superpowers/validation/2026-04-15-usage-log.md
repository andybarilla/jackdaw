# Pi workbench one-week usage log

Goal: collect enough evidence to decide whether the prototype is better than current Jackdaw habits for routine orchestration.

## How to use this log

For each real session block:
1. note how many sessions you were monitoring
2. record whether you found the session needing attention in under 10 seconds
3. count whether transcript inspection was actually necessary
4. note any intervention you made
5. document every shell fallback moment

Keep entries short and concrete.

## Baseline: current Jackdaw habit

Baseline, based on how Jackdaw normally behaves before this prototype:

- Typical number of concurrent sessions: 3-5 when things are busy
- How I usually decide which session needs attention: scan terminal panes, look for visible churn, then open the transcript or terminal output for whichever session looks suspicious
- How often I open raw terminal/transcript output for routine checks: a lot, often as the default path rather than the fallback
- Typical time to notice a blocked or waiting session: inconsistent, because the signal is buried in terminal activity instead of being surfaced as structured state
- Common reasons I need shell access: checking repo state, confirming cwd/context, and doing one-off commands when the UI does not make the session state obvious

## End-of-week rollup

Provisional rollup from the first two real usage days. This is enough to support an iterate decision, but not enough to pretend the full week happened when it did not.

- Total sessions observed across the week: 6 session-observations across 2 days
- Number of checks where the right session was identified in under 10 seconds: 6
- Routine checks handled without transcript details: 6 / 6
- Percent handled without transcript details: 100%
- Did summary + status + tool activity beat raw terminal panes for routine monitoring? yes
- Did `steer`, `follow-up`, and `abort` each feel obvious? mixed, steering clearly yes, follow-up and abort less directly validated in real use so far
- Were pinned summaries, editable names, and tags actually useful? mixed, useful enough to keep, but pinning semantics needed cleanup
- How many shell fallback moments occurred: 0 in the first two days
- Top reasons shell fallback was required: none so far
- Best example where the prototype clearly beat current Jackdaw: routine monitoring could stay in the overview instead of repeatedly opening raw transcripts just to figure out which session needed attention
- Strongest reason not to continue with this direction: the overview still had trust and clarity issues under concurrent updates, especially list flicker and muddy pinning behavior
